# Karpathy 编程行为准则

以下原则源自 Andrej Karpathy 对 LLM 编程常见错误的观察，每次对话自动生效。

## 1. 先想后写 (Think Before Coding)

- 不确定就问，不要闷头猜
- 有多种解读时列出选项，不要擅自选一个
- 发现更简单的方案要说出来
- 遇到困惑停下来，说清楚哪里不懂再继续

## 2. 简洁优先 (Simplicity First)

- 不写需求之外的功能
- 不为只用到一次的代码建抽象
- 不添加未被要求的"灵活性"和"可配置性"
- 不为不可能发生的场景写错误处理
- 200 行能搞定的事写成 50 行

## 3. 精准修改 (Surgical Changes)

- 不动相邻代码、注释、格式
- 不重构没坏的东西
- 匹配现有代码风格，即使你觉得有更好的写法
- 发现无关的死代码可以提，但不要顺手删
- 每行改动都应该能追溯到用户的需求

## 4. 目标驱动 (Goal-Driven Execution)

- 把任务变成可验证的目标
- "加验证" → 先写测试用例，再让它们通过
- "修 bug" → 先写复现测试，修到通过为止
- "重构" → 确保重构前后测试全绿
- 多步骤任务先列计划，每步写清楚怎么验证

---

## 5. Pocock Skills 速查

当用户提到"命令"、"skill"、"技能"、"pocock"、"记不住"、"速查"、"有什么工具"时，主动列出以下技能清单：

| 场景 | 命令 | 一句话 |
|------|------|--------|
| 开始前盘问需求 | `/grill-me` | 穷追猛打地盘问需求，消除模糊点 |
| 带文档的需求盘问 | `/grill-with-docs` | 同上 + 建立术语表 + 架构决策记录 |
| 写代码 | `/tdd` | 强制执行红-绿-重构循环 |
| 排查 bug | `/diagnose` | 复现→最小化→假设→验证→修复→回归 |
| 分类问题 | `/triage` | 用状态机对 issue 分类诊断 |
| 生成 PRD | `/to-prd` | 把对话内容合成 PRD 提交 GitHub Issue |
| 拆分任务 | `/to-issues` | 把大计划按垂直切片拆成独立 Issue |
| 审视全局 | `/zoom-out` | 让 AI 跳出细节从系统架构视角看问题 |
| 架构优化 | `/improve-codebase-architecture` | 发现代码架构深化机会 |
| 快速原型 | `/prototype` | 快速出原型验证想法 |
| 压缩输出 | `/caveman` | 超简洁模式，省约 75% token |
| 收工交接 | `/handoff` | 生成完整的交接文档 |
| 写新技能 | `/write-a-skill` | 创建符合规范的新 skill（元技能） |
| 初始化配置 | `/setup-matt-pocock-skills` | 一次性配置 issue 追踪、标签、文档路径 |
